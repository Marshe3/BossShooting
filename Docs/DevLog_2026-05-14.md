# 개발 일지 - 2026-05-14

## 한 줄 요약

오늘은 무기/적/웨이브/보스의 핵심 네트워크 gameplay 기반을 확장했다. 특히 서버 권위로 projectile, pickup, wave spawn, reload, boss phase를 처리하고, 클라이언트는 replicated state와 replicated movement 결과를 보는 구조를 반복해서 구현했다.

---

## 1. 작업 방식 개선

오늘 중간부터 작업 방식을 바꿨다.

기존에는 Codex가 코드를 직접 많이 수정했지만, 학습 효과를 높이기 위해 **직접 치기 모드**를 도입했다.

새 방식:

1. Codex가 전체 파일 대신 작은 코드 조각을 제공한다.
2. 사용자가 직접 입력한다.
3. Codex가 파일을 읽고 오타/UHT/RPC 문제를 확인한다.
4. Codex가 빌드로 검증한다.
5. 성공하면 학습 노트와 handoff 문서를 정리한다.

이 방식의 장점:

- 내가 어떤 파일에 어떤 코드를 넣는지 직접 따라갈 수 있다.
- Unreal C++의 `UCLASS`, `UPROPERTY`, `UFUNCTION`, replication 패턴에 익숙해진다.
- 긴 코드 전체를 붙여넣는 것보다 학습 부담이 낮다.

오늘 실제로 `EnemyWaveTrigger`, `BossQueen`, reload 기능에서 이 방식을 사용했다.

---

## 2. 코드 주석 보강

최근에 추가한 코드들에 학습용 주석을 보강했다.

주석 방향:

- 서버에서만 실행되는 로직 표시
- 클라이언트가 요청만 하는 부분 표시
- 클라이언트가 replication 결과만 보는 부분 표시
- RepNotify 수동 호출 이유 설명
- projectile spawn/damage가 서버 권위여야 하는 이유 설명

관련해서 `.agents/skills`에도 규칙을 추가했다.

앞으로 gameplay/multiplayer 코드를 만들 때는 짧은 한국어 주석으로 서버/클라 경계를 남기도록 했다.

## 3. Step 6c - 무기 픽업 기반 정리

무기 픽업 시스템의 C++ 기반을 정리하고 학습 노트를 작성했다.

핵심 구조:

- `AWeaponPickup` actor는 replicated actor다.
- `USphereComponent` overlap으로 플레이어 접근을 감지한다.
- overlap 처리는 서버에서만 한다.
- 서버가 `WeaponClass`를 spawn하고 `ABaseCharacter::EquipWeapon`을 호출한다.
- 기존 무기는 1슬롯 모델에 맞게 detach 후 destroy한다.
- pickup actor는 서버에서 destroy되고 클라이언트에서도 사라진다.

학습 포인트:

> pickup은 "누가 보상을 얻었는가"를 결정하는 gameplay state라서 서버 권위로 처리해야 한다.

## 4. Step 7a - 스피터 원거리 산성 침

두 번째 잡몹인 스피터의 C++ 기반을 만들었다.

추가한 클래스:

- `AAlienSpitter : AAlienBase`

기능:

- 러셔식 근접 공격을 끄고 원거리 공격 타이머를 사용한다.
- 서버에서만 일정 간격으로 살아있는 플레이어를 찾는다.
- 공격 범위 안에 있으면 `AcidProjectileClass`를 서버에서 spawn한다.
- 산성 침 projectile은 기존 `ABaseProjectile`을 재사용한다.

`ABaseProjectile`도 같이 보강했다.

- `DamageCauser`와 `IgnoredActor`를 분리했다.
- weapon projectile과 enemy projectile 모두 발사자를 ignore할 수 있게 했다.
- `BeginPlay`에서 BP의 `InitialSpeed`를 movement component에 다시 반영했다.

학습 포인트:

> 러셔는 서버가 직접 근접 데미지를 넣는 적이고, 스피터는 서버가 projectile actor를 spawn하는 적이다. 둘 다 공격 판단은 서버 권위지만 클라이언트가 보는 결과는 다르다.

빌드:

- `BossShootingEditor Win64 Development` 성공

## 5. Step 7b - 잡몹 웨이브 트리거

플레이어가 특정 구역에 들어가면 서버가 적 웨이브를 spawn하는 트리거를 만들었다.

추가한 클래스:

- `AEnemyWaveTrigger : AActor`

기능:

- `UBoxComponent` overlap으로 플레이어 진입을 감지한다.
- 서버에서만 overlap callback을 바인딩한다.
- 살아있는 `ABaseCharacter`가 들어오면 wave를 한 번만 실행한다.
- `EnemyClasses` 배열에 들어 있는 적 BP들을 서버에서 spawn한다.
- `SpawnPoints`가 있으면 해당 위치 기준으로 spawn한다.
- `SpawnScatterRadius`로 같은 위치에 겹치지 않게 랜덤 offset을 준다.

직접 치기 중 발생한 수정:

- `BLueprintReadOnly` 오타를 `BlueprintReadOnly`로 수정했다.
- `UPrimitiveComponent` forward declaration을 추가했다.

검증:

- PIE Listen Server에서 정상 동작 확인
- 트리거 진입 시 적이 spawn됨
- host/remote client 양쪽 화면에서 적이 보임
- 여러 번 밟아도 한 번만 실행됨

학습 포인트:

> overlap 이벤트는 단순하지만, enemy spawn은 게임 상태 변경이므로 서버가 결정해야 한다.

빌드:

- `BossShootingEditor Win64 Development` 성공

작성 문서:

- `Docs/Step7b_Enemy_Wave_Trigger_Notes.md`

## 6. Step 8a - BossQueen 페이즈 상태 기반

보스 공격 패턴 전에 보스전의 중심 상태인 phase replication 기반을 만들었다.

추가한 클래스:

- `ABossQueen : AAlienBase`

추가한 enum:

- `EBossQueenPhase`
  - `Phase1`
  - `Phase2`
  - `Phase3`
  - `Dead`

기능:

- 보스 체력은 `MaxHealth = 600`으로 설정했다.
- 기존 `AAlienBase`의 health/damage/death 흐름을 재사용한다.
- HP 60% 이하에서 Phase2로 전환한다.
- HP 30% 이하에서 Phase3로 전환한다.
- 사망 시 Dead phase로 전환한다.
- `CurrentPhase`는 `ReplicatedUsing=OnRep_CurrentPhase`로 복제한다.
- `bIsInvincible`도 replicated state로 추가했다.

직접 치기 후 정리한 부분:

- include 경로를 `Enemy/AlienBase.h`로 정리했다.
- `BOss|Phase` 오타를 `Boss|Phase`로 수정했다.
- 자동 생성된 불필요한 `Tick`, `SetupPlayerInputComponent` 선언/구현을 제거했다.
- `bIsInvincible`일 때 데미지를 무시하는 체크를 추가했다.

검증:

- C++ 빌드 성공
- BP 에셋 생성 및 PIE 검증은 다음 단계로 남음

학습 포인트:

> 보스 phase는 단발 이벤트가 아니라 지속 상태다. 따라서 multicast만 쓰기보다 replicated property로 들고 있는 편이 안전하다.

빌드:

- `BossShootingEditor Win64 Development` 성공

## 7. Step 6d - 무기 장전

보스 작업 전에 무기 장전 기능을 먼저 추가했다.

수정한 클래스:

- `ABaseWeapon`
- `ABaseCharacter`

기능:

- `ReloadDuration` 튜닝값 추가
- `bIsReloading` replicated state 추가
- `StartReload()` 입력 진입점 추가
- `Server_Reload` Reliable RPC 추가
- 서버가 장전 가능 여부를 판단한다.
- 서버 timer가 끝나면 `CurrentAmmo = MaxAmmo`로 복구한다.
- 장전 중에는 서버에서 발사를 거부한다.
- `ABaseCharacter`에 `ReloadAction`, `OnReloadPressed()`를 추가했다.

직접 치기 후 정리한 부분:

- `ClampMIn` 오타를 `ClampMin`으로 수정했다.
- `FTimerHandle` 사용을 위해 `TimerManager.h`를 추가했다.
- 장전 중 발사 거부를 클라이언트 쪽이 아니라 서버 `Server_Fire`에서 확실히 처리하도록 옮겼다.
- 무기 해제 시 reload timer를 정리하도록 했다.
- 장전 완료 시 무기 소유자가 사라진 경우 안전 처리를 추가했다.

검증:

- C++ 빌드 성공
- BP에서 `IA_Reload` 생성, `IMC_Default` R키 매핑, `BP_BaseCharacter.ReloadAction` 할당
- PIE Listen Server에서 장전 동작 정상 확인
- 장전 중 발사 거부 및 장전 완료 후 탄약 복구 확인

학습 포인트:

> 장전은 클라이언트 입력으로 시작하지만, 탄약을 채우는 최종 결정은 서버 timer와 replicated ammo state로 처리해야 한다.

빌드:

- `BossShootingEditor Win64 Development` 성공

## 8. 오늘 반복해서 학습한 네트워크 패턴

오늘 여러 기능에서 같은 서버 권위 패턴을 반복했다.

### 클라이언트 요청, 서버 결정

예시:

- 발사
- 장전

클라이언트는 입력만 보낸다. 서버가 탄약, 쿨다운, 장전 상태를 판단한다.

### 서버 spawn, 클라이언트 관찰

예시:

- 유탄 projectile
- 산성 침 projectile
- 웨이브 적 spawn

서버가 actor를 만들고, 클라이언트는 replication으로 actor와 movement를 본다.

### RepNotify로 상태 반응

예시:

- `CurrentAmmo`
- `bIsReloading`
- `CurrentPhase`

값은 replication으로 전달되고, 클라이언트는 `OnRep`에서 HUD/연출 hook을 붙일 수 있다.

### 서버 내부 guard는 굳이 replicate하지 않음

예시:

- `AWeaponPickup::bClaimed`
- `AEnemyWaveTrigger::bTriggered`

클라이언트가 직접 읽을 필요 없는 서버 내부 중복 방지 값은 replicate하지 않았다.

## 10. 오늘의 회고

오늘은 “기능을 하나씩 추가했다”기보다, 같은 네트워크 원칙을 여러 gameplay 상황에 반복 적용한 날이다.

가장 중요한 흐름은 다음과 같다.

```text
클라이언트 입력 또는 overlap
-> 서버가 유효성 판단
-> 서버가 gameplay state 변경 또는 actor spawn
-> 클라이언트는 replication / RepNotify / multicast debug FX로 결과 확인
```

이 패턴을 장전, projectile, 웨이브, 보스 phase에 모두 적용했다.

또한 직접 치기 모드를 도입하면서 코드 구조를 더 능동적으로 읽게 되었다. 오타는 생겼지만, UHT/빌드가 어떤 부분에서 민감하게 반응하는지도 직접 확인했다. 앞으로도 작은 단위로 직접 치고, 빌드와 학습 노트로 마무리하는 방식이 이 프로젝트의 학습 목적에 잘 맞는다.

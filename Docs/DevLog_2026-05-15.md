## 2. Step 8b - BossQueen Phase1 공격 패턴

Phase1에서 보스가 기본 공격을 하도록 만들었다.

구현한 패턴:

- 가까운 플레이어가 있으면 전방 부채꼴 근접 공격
- 먼 플레이어가 있으면 projectile 3방향 발사
- 공격 판단은 서버 타이머에서 실행
- 근접 데미지와 projectile spawn은 서버에서 처리
- 클라이언트는 replicated health, projectile actor, multicast debug FX 결과를 본다

수정 파일:

- `Source/BossShooting/Public/Enemy/BossQueen.h`
- `Source/BossShooting/Private/Enemy/BossQueen.cpp`

중간 문제:

처음에는 근접 공격이 한 번만 들어가는 것처럼 보였다.

원인:

- 근접 공격은 단순 거리 판정이 아니라 전방 부채꼴 판정이었다.
- 보스가 공격 직전에 플레이어를 바라보지 않으면, 가까이 있어도 각도 판정에서 빠질 수 있었다.

해결:

- `FaceTarget_ServerOnly(ABaseCharacter* Target)` 추가
- 공격 직전에 서버에서 보스가 타겟을 바라보게 함
- 회전은 movement replication으로 클라이언트에도 보임

검증:

- 가까이 붙으면 근접 공격이 반복해서 들어감
- 멀리 있으면 3방향 projectile 발사
- host/remote client 양쪽에서 같은 결과 확인

## 5. Projectile 폭발 remote client 미표시 문제 해결

오늘 가장 중요한 트러블슈팅 중 하나였다.

증상:

- projectile actor 자체는 remote client에서도 보임
- 하지만 충돌 후 폭발 debug sphere가 remote client에서 안 보임
- 데미지는 정상 적용됨

기존 코드 흐름:

```cpp
Multicast_Explode(ExplosionLocation);
Destroy();
```

문제:

- `Multicast_Explode`가 `Unreliable`
- multicast 호출 직후 projectile actor를 바로 `Destroy`
- remote client가 actor destroy를 먼저 처리하거나 unreliable RPC를 놓치면 폭발 연출을 못 볼 수 있음

수정:

- `Multicast_Explode`를 `Reliable`로 변경
- 폭발 직후 바로 destroy하지 않음
- collision 비활성화
- projectile movement 정지
- `SetLifeSpan(0.2f)`로 짧게 남긴 뒤 제거

수정 후 흐름:

```text
서버가 radial damage 적용
collision 끔
movement 정지
Reliable multicast로 폭발 연출 전파
0.2초 뒤 actor 제거
```

검증:

- host가 쏜 런처 폭발이 remote client에서도 보임
- 적/보스 원거리 projectile 폭발이 remote client에서도 보임
- 데미지 정상 적용

작성 문서:

- `Docs/Troubleshooting_Projectile_Explosion_Multicast_Remote_Client.md`

빌드:

- `BossShootingEditor Win64 Development` 성공

학습 포인트:

> replicated actor가 보인다고 해서, 그 actor가 죽기 직전에 보낸 unreliable multicast까지 항상 보장되는 것은 아니다. Destroy 직전 one-shot FX는 Reliable multicast 또는 짧은 destroy delay가 필요할 수 있다.

## 6. Projectile 시각화에 대한 결정

원거리 공격이 너무 갑자기 맞는 느낌이 있어서 projectile 자체를 눈에 보이게 하는 방법도 정리했다.

현재 구조:

```text
CollisionComponent = 실제 충돌/판정
ProjectileMesh 또는 Niagara = 시각 요소
```

권장 BP 작업:

- `BP_AcidProjectile`에 `StaticMeshComponent` 추가
- 임시 mesh는 `/Engine/BasicShapes/Sphere`
- scale은 `0.2`
- material은 초록/노랑 emissive 추천
- visual mesh collision은 `NoCollision`
- 실제 판정은 부모의 `CollisionComponent`가 담당
- 테스트 중 projectile이 너무 빠르면 `InitialSpeed`를 낮춤

이렇게 하면 "폭발 반경만 갑자기 보이는 공격"에서 "날아오는 projectile을 보고 피할 수 있는 공격"으로 바뀐다.

## 7. 오늘의 네트워크 패턴 정리

오늘 반복해서 사용한 원칙:

```text
Gameplay result = server authoritative
Persistent state = replicated property
One-shot visual feedback = multicast
Owner input = request only
```

적용 예:

- Boss phase: `CurrentPhase` replicated
- Boss invincible: `bIsInvincible` replicated
- Boss attack timer: server only
- Enemy summon: server spawn
- Projectile movement: replicated actor movement
- Projectile damage: server radial damage
- Projectile explosion visual: Reliable NetMulticast

오늘 특히 배운 점:

- 서버에서 actor를 spawn하면 클라이언트는 actor replication으로 본다.
- 하지만 actor가 사라지기 직전에 보내는 one-shot multicast는 놓칠 수 있다.
- 공격 판정이 수학 기반이면, range뿐 아니라 direction/angle 조건도 디버깅해야 한다.
- 타이머는 gameplay 판단 주기를 명확히 만들 수 있지만 phase 전환/사망 시 반드시 정리해야 한다.

## 9. 오늘의 회고

오늘은 단순히 기능을 추가한 날이라기보다, **멀티플레이에서 “보이는 것”과 “실제로 적용되는 것”이 다를 수 있다**는 걸 직접 확인한 날이었다.

projectile 데미지는 서버에서 잘 적용되고 있었지만, remote client에서 폭발이 안 보이니 플레이어 입장에서는 일방적으로 맞는 느낌이 났다. 이 문제를 고치면서 gameplay state와 cosmetic feedback을 분리해서 생각하는 감각이 더 선명해졌다.

BossQueen도 마찬가지다. Phase, 무적, 소환, 공격 타이머, projectile spawn은 전부 서버가 책임지고, 클라이언트는 결과를 본다. 이제 보스전의 큰 뼈대는 거의 잡혔다.

다음부터는 이 뼈대 위에 HUD, 연출, 난이도 튜닝을 얹으면 게임처럼 보이는 단계로 넘어갈 수 있다.
